Two mid-loop lead timeouts (900 s) emitted truncated drafts (rounds 1 and 4 — 2.7k / 8.7k chars), which poisoned the convergence trajectory (blocking findings 10→9→8→2→11); round 5's full 66k draft is the landed base. ## Residual blockers — hand-resolved in the landed plan

1. **No executable `.app` build** → replaced the xcodebuild-scheme claim with `console/scripts/assemble-app.sh` (swift build -c release → explicit bundle assembly → ad-hoc codesign), declared bundle path, `AppBundleIdentityTests` runs the script and verifies plist/executable/codesign.
2. **P1 consumed `Settings` (produced in P4)** → Phase 1 owns a dependency-free `ResolutionInputs`; `Settings.resolutionInputs()` maps into it in P4. Top-down compile order restored.
3. **Validator keyword under-coverage** → `items`, `minLength`, `maxLength`, `pattern` added to the implemented set + semantic negatives; `SchemaKeywordCoverageTests` now walks every bound schema and fails on any unimplemented keyword.
4. **Unknown-event vs closed union** → decode order pinned: envelope subschema first, known case → full union member validation, unknown discriminator → `.unknown(raw:)` without applying the closed union.
5. **No stderr on `StreamHandle`** → `completion()` returns `StreamCompletion{exitCode, stderr}`; watch stderr is captured and surfaced, never swallowed.
6. **Disconnected attach/cursor sequence** → new **P4-Task-5 `AttachCoordinator`** (once-hello → incarnation lookup → resume plan → live spawn → safe-heartbeat checkpoints, sole checkpoint writer) + named `AttachResumeCheckpointFlowTests`.
7. **`authorizeArgv` lacked operands** → `BoundInvocation` (immutable op + fully-resolved operand argv); export and authorize argv derive from the same value, making AuthorizeRetry's same-argv guarantee structural.
8. **Hardcoded `BackstopEligibleCommands`** → deleted. The exit-6 backstop reuses the refused invocation's own argv verbatim (`handleExit6(refused: BoundInvocation, …)`) — works for any mutation, no second classification authority, nothing to retire when atlas fixes the git-refresh drift.
9. **`OperandSourceMap` authority** → scoped to `authorizableOps` only, drift-guarded by `operandSourceMapCoversAuthorizableOps`, and **authorized by a spec amendment on this PR** (the spec's ssot section now records three Console-owned exceptions); atlas-side operand `source` field recorded as the retiring follow-up.
10. **No composition root** → new **P6-Task-4** (`AtlasConsoleApp`, `AppModel` launch sequence with blocking probe state, navigation, `SettingsView`, settings-change re-probe) + `AppLaunchProbeTests` / `SettingsReprobeTests`.
11. **No generic read executor** → `DetailReader` replaced by `ReadCommandExecutor` (execution-class-gated, schema-bound, strict success/error parsing, the single gateway for the runtime-inventoried 25-command read surface) + `ReadCommandExecutorTests` covering each execution class.

### Round 1 — verdict: revise (374s, draft 2677 chars)

**Summary:** The architecture is scope-appropriate and largely covers the spec, but circular foundation dependencies, incomplete subprocess and argv contracts, missing app composition/settings work, and non-runnable verification prevent execution as written.

**Blocking (10):**
- Phase 1 Tasks 3–4 (lines 77–88) form an implementation cycle and depend on Phase 2: executable resolution consumes `ProcessRunner` and the P2 db-status decoder, while `ProcessRunner` consumes `ResolvedBinary` and P2's `EgressMintingCommands`. Split out a dependency-free spawn executor usable by probes, move probe validation and the egress constant to P1, or reorder the contracts so P1 can complete without later-phase outputs.
- Phase 1 Task 4's `ProcessRunner` interface (line 87) cannot support Phase 5 Task 3: it exposes neither stdin bytes nor a per-invocation cwd, yet signing must pipe frozen challenge bytes to stdin and privileged flows must run in a 0700 temp directory. Declare these exact inputs in the produced interface and test them before P5 consumes it.
- Phase 5 Task 1's interface (line 264) declares `authorizeArgv(op:, authPath:)` without the original operation inputs. Authorizable commands such as `source trust promote <sourceId>` and `db restore <backupRef>` require the same operands during export and authorization, so the declared signature cannot assemble a valid authorize invocation or preserve exact argv for `AuthorizeRetry`; produce a bound invocation or add `inputs:` to the authorize signature.
- Phase 2 Task 3 (line 124) produces `[OpDescriptor stub]`, while the real `OpDescriptor` is introduced only in Phase 5. `stub` is a prohibited placeholder and creates a cross-phase type mismatch. P2 should produce exact registry command identifiers plus parsed contract metadata; P5 should consume those and produce the finalized descriptor type.
- Phase 2's strict-schema claim (lines 100–140) does not specify an executable validation mechanism. Swift `Codable` alone ignores unknown properties and does not enforce JSON-Schema constraints such as `unevaluatedProperties:false`, ranges, consts, and unions. Name the pinned Swift dependency or generator/custom validator, its supported schema dialect/keywords, and tests proving extra properties and constraint violations fail.
- Phase 2 Task 5 and Phase 4 Task 5 (lines 135–140 and 238–243) use the prohibited `…` placeholder and declare typed interfaces for only a few reads, despite the required runtime-inventoried 25-command non-mutating surface. Define a generic schema-bound read executor covering every inventoried `pure`/`read`/`audited-read` command and explicitly name the typed UI adapters that V1 exposes.
- No phase owns the runnable application composition root or a process-settings UI. P1 creates an app target and stores settings, while P4 creates disconnected views; no exact task wires launch probing, supervisor lifecycle, reducers, navigation, settings edits/re-probes, or the controls for `atlasRoot`, path overrides, cadence, resume mode, and egress-key source. Add an exact `AtlasConsoleApp`/app-model/settings-view task with interfaces and tests.
- Phase 5 Task 2 consumes 'the SP-3 challenge schema (from bundle)' (line 271), but P1's `ContractBundle` is explicitly only `commands.json` plus CLI-contract schemas, and the current tree has no `console/signer/` implementation. Define the exact SP-3 challenge/response schema source and how it is bound to the resolved signer; keep the existing prerequisite as a hard preflight gate before P1.
- Phase 6 Task 1 only creates tests and a checklist (lines 306–311); no behavior-changing task implements live announcements, subprocess busy/completion announcements, modal focus entry/restoration, or reduced-motion behavior. Assign those behaviors to exact P4/P5 source files and name their proving tests, leaving P6 as the acceptance gate.
- The Verification sections for Phases 2–6 (lines 146–147, 198–199, 249–250, 294–295, and 337–338) list test numbers or manual goals rather than commands that run as written. Provide exact `swift test --package-path console --filter ...`, headless-engine/app launch commands, and exact live-drive invocation/evidence commands for every phase.

**Non-blocking (1):**

### Round 2 — verdict: revise (1406s, draft 82058 chars)

**Summary:** The plan is scope-appropriate and close to executable, but unresolved stream, validation, descriptor, cursor, and phase-boundary contracts still require revision.

**Blocking (9):**
- Phase 1 Task 3, Phase 2 Task 4, and Phase 3 Task 1 declare incompatible streaming contracts: `Spawn.stream` yields already newline-framed lines, while `NDJSONFramer` expects arbitrary byte chunks and tests split lines/UTF-8 scalars. The stream interface also exposes no completion result carrying the process exit code and stderr, so `WatchSupervisor` cannot classify exits 0/2/4/5 as required. Produce a stream handle with raw byte chunks plus an awaited `SpawnResult`, or move framing into `Spawn` and adjust the downstream contract/tests.
- Phase 1 Task 6's brain probe only performs a minimal required-key check, but the spec requires `brain db status --json` to produce schema-valid output. Nothing later revalidates the probe against the bound db-status schema, so malformed output with the required three keys would incorrectly pass. Make full schema validation part of the launch/settings-change probe.
- Phase 1 Task 6 produces `SignerContract` as fixture arrays plus signature algorithm names, but Phase 5 Tasks 2-3 consume an undeclared challenge/response validation shape. Embedded examples are not a validator and do not define required keys, unknown-key rejection, field constraints, or the response type returned after validation. Declare exact challenge/response model or validator interfaces and their strict negative tests.
- Phase 2 Task 1's claimed schema subset is factually incomplete for the consumed CLI schemas, which use constraints including `minLength`, `maxLength`, `maxItems`, `pattern`, `items`, and `properties`. It also duplicates schema bounds/enums inline in Swift, so schema value changes can drift without the keyword scanner noticing. Phase 2 Task 6 then invokes `StrictDecoder` generically as `ValidatedJSON`, although neither runtime schema evaluation nor the `ValidatedJSON` type is produced. Replace this with an executable runtime or generated schema-validation contract covering every consumed keyword and exact constraint value.
- Phase 2 Task 5 and Phase 5 Task 1 claim `Operand.source`—focused UI object versus operator entry—is parsed from `x-atlas-contract`, but current command schemas describe args and flags without such a `source` field. The plan therefore cannot construct the promised descriptors as written. Declare the exact Console-owned routing metadata and its SSOT/drift checks, or add the required atlas-side contract field before consuming it.
- Phase 3 Task 3 specifies cursor persistence only at the first post-replay checkpoint heartbeat. The spec also requires `liveOnly` and first-sight live streams to checkpoint progress so a later `resume` works. Explicitly consume every safe attached heartbeat after replay, transactionally persist its contiguous run-space prefix, and name the test proving live-only progress survives restart.
- Phase 4 has a back-edge into Phase 5: Task 3 consumes the later `FlowState`, and Task 1 hosts the Phase-5 privileged-flow sheet/query surface, while Phase 5 declares P4 as a dependency. Phase 4 therefore cannot compile as its promised working increment. Define a phase-independent announcement/event interface in P4 and defer privileged-flow composition/navigation until P5.
- Phase 6 Task 4 still contains the prohibited placeholder path `docs/retros/2026-07-XX-console-live-drive-retro.md`. Replace it with the finalized exact artifact path.
- Verification commands are not all executable as written: Phase 1 writes `brainPathOverride` as a top-level defaults key even though Task 5 stores Settings as one blob, and Phase 6 step 4 uses the invalid launchctl target `system/`. Provide commands matching the actual persisted blob/key contract and a valid `launchctl bootstrap system <plist>` invocation.

### Round 3 — verdict: revise (577s, draft 98444 chars)

**Summary:** The revision resolves the prior stream and process-contract gaps, but schema coverage, signer shape fidelity, Swift target/app packaging, reducer initialization, and exit-6 routing still prevent execution as written.

**Blocking (8):**
- Phase 1 Task 6: the declared SchemaValidator keyword set omits `allOf`, `if`, `then`, and `not`, although the current consumed schemas use them extensively, including `query.schema.json`, `purge.schema.json`, and `index-repair.schema.json`. `SchemaKeywordCoverageTests` will therefore fail in P1, and adding the keywords without implementing their semantics would under-validate responses. Add these keywords and positive/negative semantic tests, or use a conforming Draft 2020-12 validator.
- Phase 1 Task 7: `AuthorizationChallenge` and `SignerContractValidator.validateChallenge` omit the required `schemaVersion` and `payloadCanonicalization` fields from security-broker-contract.md §7.1. The response validator also declares `challenge: JSONValue` without requiring nested validation as an AuthorizationChallenge. Define both required fields and recursively validate the echoed response challenge against the same strict challenge contract.
- Phase 1 Task 1 and Phases 4–6: the plan creates only a SwiftPM executable but later assumes a macOS application bundle with bundle identifier `com.atlas.console`, `UserDefaults.standard` addressable through `defaults write com.atlas.console`, ad-hoc signing, and an XCUI launch target. `swift build`/`swift run` alone does not declare that bundle identity or an XCUI-hostable `.app`. Add exact app-bundle/Info.plist assembly and signing files plus an executable UI-test launch contract, or change the defaults and accessibility verification to mechanisms that work with the actual SwiftPM executable.
- Phase 4 Tasks 1–5 and Phase 5 Task 3 create Swift target dependency cycles: `AtlasConsole` must depend on `ConsoleUI` and `ConsoleCore`, but `SettingsView` and the other ConsoleUI views consume `AtlasConsole.AppModel`, while `ConsoleCore.PrivilegedFlowCoordinator` also consumes `AppModel`. Library targets cannot depend back on their executable consumer. Move shared model/protocol state into a lower-level module, make views consume bindings/protocols/closures, and perform coordinator registration only in the AtlasConsole composition target.
- Phase 3 Tasks 1–3 do not declare how `AuditReducer` seeds its contiguous-prefix baseline on `watch.hello`. For a live-only attach to an existing ledger whose hello reports prefix 100, the first live audit row 101 appears to follow a 0–100 gap, so `checkpointSeq()` cannot advance and later resume is broken. Add an exact `reset/incorporateHello(baselinePrefix:)` interface using `hello.resume.auditHeadSeq` with replayAll’s `-1` semantics, and name a test for the existing-ledger live-only case.
- Phase 5 Tasks 1 and 3 contradict each other on the exit-6 backstop. Task 1 constructs descriptors and operand-source mappings only for registry-derived `authorizableOps`, while Task 3 promises that a registry-shared command such as `git refresh` can enter Export after broker exit 6. Such an op has no descriptor/map under the declared interfaces and will become unsupported instead of entering the signer flow. Define descriptor construction and operand-source coverage for every surfaced command that can trigger the backstop, including `git refresh`, without adding it to the registry-derived privileged set.
- Phase 1 ordering is not top-down executable: Task 4 `ProcessRunner` consumes `Settings` from Task 5 and `ResolvedBinary` from Task 7. Reorder the phase so Settings, SchemaValidator, and ExecutableResolver are produced before ProcessRunner, or split the shared input types into an earlier task.
- Prerequisites §2.2 is over-engineering and an unnecessary execution gate: it requires running launchd daemons and an enrolled Secure-Enclave signer before any P1 scaffolding or unit work, even though P1–P5 explicitly use stubs/software fixtures and the spec reserves real daemons/SE for the live drive. Move those requirements to Phase 6’s live-drive preflight so implementation and CI can begin on an unprovisioned macOS development machine.

**Non-blocking (1):**
- Clarify that the generic single-object ReadExecutor invokes the registry `watch` command only with `--once`, since bare `brain watch --json` is long-lived NDJSON rather than a one-object read.

### Round 5 — verdict: revise (420s, draft 66428 chars)

**Summary:** The plan is scope-appropriate, but unresolved package, schema, attach-state, privileged-routing, and application-composition contracts still prevent top-down execution.

**Blocking (11):**
- P1-Task-1 does not define an executable macOS app-bundle build. A SwiftPM executable plus `Resources/Info.plist` and `xcodebuild -scheme AtlasConsole` does not itself establish an application target that emits an XCUI-launchable `.app` with bundle identifier `com.atlas.console`. Declare the exact Xcode application target/project or concrete bundle-assembly/signing command and the resulting bundle path.
- P1-Task-5 consumes `Settings`, but `Settings` is not produced until P4-Task-2. Phase 1 therefore cannot compile top-down. Move the settings model to Phase 1 or change resolution to consume an earlier dependency-free configuration type.
- P1-Task-4's advertised JSON-Schema keyword coverage is incomplete. The bound schemas currently use `items`, `minLength`, `maxLength`, and `pattern`, none of which appear in the validator contract. Strict validation can therefore accept invalid command output. Implement every consumed keyword and add semantic negative tests for them, or use a conforming Draft 2020-12 validator.
- P2-Task-1 says every event is first validated against `watch.schema.json`, while also promising `.unknown` for a future event discriminator. The current schema's closed union rejects an unknown discriminator, so the unknown-event test cannot pass as described. Inspect and preserve the common envelope before applying known-case schema validation, with an explicit unknown-event path.
- P1-Task-3 exposes only stdout chunks and an exit code for streaming processes. It exposes no captured stderr, so P2-Task-5 and the UI cannot satisfy the requirement that watch stderr is never swallowed and is surfaced on the relevant error state. Add stderr capture to `StreamHandle` or its completion result.
- P3-Task-4, P4-Task-1, and P4-Task-3 leave the attach/cursor sequence disconnected. `WatchSupervisor` neither consumes `CursorStore` nor owns the initial `brain watch --once` attach needed to obtain `ledger.path` before loading the incarnation cursor; no interface threads safe attached heartbeats into transactional checkpoints. Add one coordinator implementing once-hello → incarnation lookup → resume planning → live spawn → safe-heartbeat checkpoint, with a named integration test.
- P5-Task-1 still declares `authorizeArgv(_ op:authorizationPath:)` without the operands used during export. Operand-bearing operations cannot be authorized correctly, and AuthorizeRetry cannot guarantee the exact same argv. Produce a bound immutable invocation containing the resolved operands and derive both export and authorize argv from it.
- P5-Tasks 1 and 3 introduce hardcoded `BackstopEligibleCommands`, including `git refresh`. This is a second classification/routing authority and only handles a selected subset, contradicting the requirement that any mutation returning exit 6 enters Export. Build descriptors on demand from the surfaced command's bound schema, or fail explicitly if the schema cannot route it, without a hardcoded command-membership list.
- P5-Task-1 claims operand types and focused-object/operator-entry sources are schema-derived, but the declared `Operand` has no type and the current `x-atlas-contract` metadata does not provide a machine-readable source field. `OperandSourceMap` is a Swift-side workaround for a contract gap that the spec requires to be handled by an atlas-side additive contract change. Make that contract addition a prerequisite or revise the descriptor contract to an explicitly spec-authorized owner.
- Phase 6 has individual views but no task owning `AtlasConsoleApp`, application model, navigation, launch probing, supervisor/reducer wiring, settings editing, or settings-change re-probing. Consequently the process-config surface and runnable cockpit are not implemented. Add exact composition-root and settings-view files, interfaces, and named launch/re-probe tests.
- P6-Task-1's `DetailReader` does not consume `ContractBundle` or the runtime execution-class inventory, and P5-Task-5 only tests three representative reads. No task therefore implements the required generic schema-bound executor for the runtime-inventoried 25-command read surface. Declare that executor, including schema lookup, strict success/error parsing, execution-class gating, and its consumers.
