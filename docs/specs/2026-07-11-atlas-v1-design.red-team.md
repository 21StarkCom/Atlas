## Synthesis

The main tension is between hard security/DR guarantees and deployable local ergonomics: separating every capability and failing closed protects the vault, but it creates recovery and setup paths that must be explicit. A second tension is between Markdown-as-SSOT purity and immediate safety gating: provenance state serialized through Markdown is auditable, but some derived gates must react before multi-note ChangePlans finish. ## Findings

### 🔴 `rt1` — security-trust (high)

**Concern.** The spec allows the egress broker and integration broker to “share the broker OS identity,” collapsing the network/credential boundary with the protected-ref writer boundary.

**Consequence.** The egress broker handles untrusted prompts, retrieval context, and provider responses. If that process runs as the same uid that owns protected refs, ordinary file permissions cannot prevent canonical/audit ref mutation after an egress-side compromise.

**Counter-proposal.** Forbid shared OS identity in V1: run egress and integration brokers as separate users/keychain access groups; egress owns only provider credentials/network, integration owns only protected refs/WORM anchor, with narrow validated IPC between them.

**Trade-off.** More install complexity, more IPC, and separate platform-specific setup tests.

### 🟡 `rt2` — reliability-distsys (medium)

**Concern.** The restore contract only says `brain db restore <backupRef>` “restores the ledger tables transactionally, then triggers a projection rebuild + index rebuild”; it does not bind the backup to the canonical branch/audit head at that ledger sequence.

**Consequence.** A run can integrate canonical state, write ledger rows, then fail backup verification, leaving the latest verified backup behind current Markdown/git. Restoring that backup loses the primary ledger explanation for already-present canonical commits unless an undeclared audit roll-forward happens.

**Counter-proposal.** Make every backup carry ledger seq, canonical HEAD, audit head seq/hash, and restore mode. Default restore rejects drift unless it rolls forward from signed audit events to current head or explicitly restores canonical/audit refs to the backed-up heads under a privileged time-travel challenge.

**Trade-off.** Restore is more complex and may fail more often when refs drift; point-in-time restore needs retained git/audit material.

### ⚪ `rt4` — product-dx (low)

**Concern.** The CLI synopsis advertises `brain purge ... (--yes | --authorization <file> | --export-challenge)` even though the purge contract says `--yes` alone never authorizes it.

**Consequence.** A user or implementer can reasonably read `--yes` as an authorization alternative for the most destructive command. That can produce failed first-contact usage or, worse, schema/tests that accidentally encode `--yes` as sufficient authorization.

**Counter-proposal.** Move `--yes` outside the authorization grammar and make the schema reject `--apply --yes` unless a broker-verifiable OS-presence assertion or `--authorization` is also present.

**Trade-off.** Help text is more verbose and the interactive presence path must be named explicitly.

### 1. Separate broker identities

**Addresses:** `rt1`
**Sections touched:** `Security & privacy / Egress guard`, `Git workflow / Authorization boundary`, `Model provider layer`, `Phased build plan`, `Testing / Approval-boundary + broker`

**Rationale.** Require distinct OS users and key custody for egress and integration. The egress broker owns provider credentials and outbound network only; the integration broker owns protected refs, approval verification, and the WORM audit anchor only. Any interaction between them must cross a narrow validated IPC boundary, so an egress compromise cannot mutate canonical or audit refs via shared filesystem permissions.

**New trade-off.** Installation and CI become more complex because two platform-specific identities, keychain scopes, and IPC paths must be provisioned and tested.

### 2. Bind restores to ref state

**Addresses:** _None_
**Sections touched:** `SQLite tables / Two classes of state`, `Ledger backup subsystem`, `Git workflow / Audit SSOT`, `Testing / Ledger backup/restore`, `Testing / Audit disaster-recovery`

**Rationale.** Make each ledger backup carry the ledger sequence plus the canonical HEAD and audit head sequence/hash it covers. Restore must choose an explicit mode: reject drift by default, roll forward from verified signed audit events when possible, or perform privileged point-in-time restore of canonical/audit refs to the backed-up heads. This prevents restoring a ledger that no longer explains current Markdown/git state.

**New trade-off.** Restore is less automatic: drift can block recovery until the operator chooses roll-forward or privileged point-in-time restore, and point-in-time restore depends on retained git/audit material.
