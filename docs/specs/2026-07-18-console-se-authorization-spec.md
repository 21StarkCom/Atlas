# Atlas Console SP-3 — Secure-Enclave authorization signing: the P-256 signer class, `atlas-signer`, and per-device enrollment

> **Status:** proposed · **Version:** 1 · **Arc:** Atlas Console SP-3 (of SP-1 enablement / SP-2 SwiftUI cockpit / SP-3 privileged signing)
> **Decision record:** [`docs/adr/0002-p256-secure-enclave-authorization-signer.md`](../adr/0002-p256-secure-enclave-authorization-signer.md) (immutable; this spec carries the full design).
> **Consumes:** the §7 challenge/response protocol, §7.5 `authzContract`, §8 envelope/canonicalization, §9 signer registry, §10 rotation ([`security-broker-contract.md`](security-broker-contract.md)); the D20 test-signer gate (plan §2.6, `packages/broker/src/authorize.ts`); the provisioning key-custody layout ([`provisioning/CLAUDE.md`](../../provisioning/CLAUDE.md), `keys.acl.json`); the design SSOT's *Authorization boundary* (presence vs informed approval, lines on trusted display).
> **Produces:** the design for algorithm-agile authorization verification (the normative rules land in `security-broker-contract.md` via the §5.3 amendment — that file, not this one, is the post-merge operational SSOT), the `atlas-signer` tool, Secure-Enclave key custody, and the per-device enrollment/revocation flow. SP-2 consumes `atlas-signer` as its only signing path.

Atlas Console must be able to drive the privileged, broker-authorized operations — `graduation migrate --apply/--rollback`, `purge`, `db restore`, `git approve`/`git rollback`, `git refresh`, `source trust promote`/`revoke`, `quarantine inspect`/`quarantine resolve` (the full `SIGNATURE_AUTHORIZABLE_OPS` set, §5.3) — without ever becoming a credential holder itself. Today the non-interactive signing flow exists (`--export-challenge` → sign → `--authorization`) but **no production signer is enrolled**: provisioning seeds `approval-verify.pub` empty, and the only working signer is the D20-gated test fixture. SP-3 fills that hole with the strongest custody this hardware offers — a Secure-Enclave-born, Touch-ID-gated P-256 key — and the small amount of algorithm agility the Ed25519-only envelope needs to accept it.

---

## 1. Problem

The privileged surface is designed around "a separately held key the agent process cannot read" (design SSOT, *Authorization boundary*). On this single-operator Mac install, the realistic instantiations are:

- **A file key** (Ed25519 PEM somewhere the agent UID can't read). Works, but it is ambient: any process running as the *operator* can sign silently, in bulk, forever. No per-use human act.
- **A Secure-Enclave key.** Enclave-born and non-exportable (the OS holds only an encrypted blob usable solely by this Mac's SE), and each signature can be gated on biometry — a physical human act per approval. This is categorically better custody — and it is **P-256-only**, which the current Ed25519-pinned envelope cannot verify.

Meanwhile the Console (SP-2) needs a *click-path* to privileged ops. Bridging its UI to a file key would quietly downgrade the strongest link in the system at exactly the moment it gets a GUI. The right order is: make the envelope accept a P-256 signer class (this spec + ADR-0002), put the key in the enclave, and give both the terminal and the Console one shared signing tool.

## 2. What this is

- **ADR-0002 applied**: `alg: "ed25519" | "p256"` on signer-registry entries (absent ⇒ ed25519), `p256:<base64url(DER)>` signatures over the same §8.2 `signingPayload` bytes, SPKI-PEM public-key interchange, verify dispatch in the broker. Closed set, no negotiation.
- **`atlas-signer`** — a standalone Swift CLI (SwiftPM, no GUI, ad-hoc-signed, built from source) that owns the SE key end-to-end: `keygen`, `pubkey`, and `sign` (read challenge JSON on stdin → display the exact operation and commitments → Touch ID → `AuthorizationResponse` JSON on stdout). Drop-in symmetric with `tools/test-signer.ts`.
- **Per-device enrollment/revocation** — a provisioning-owned, `sudo`-gated script that installs the exported public key as a `signers.json` registry entry (`signerId` convention `approver-se-<hostname>-vN`), and revokes by flipping `status`. The fresh-Mac story and the biometry-re-enrollment story are the same flow.
- **The `os-presence` mechanism becomes implementable** — registry entries gain a `presence` claim; presence-gated signers (and only they) may hold the two quarantine ops in `permittedOps` (§7.1). This *is* a change to which signers may authorize which ops — owned here, not smuggled.
- **D20 extended to the new algorithm** — the fixture-signer hard-reject becomes a set covering the CI software-P256 signer; no fixture signer of any algorithm can authorize outside `ATLAS_TEST_MODE`.
- **An honest threat model** (§9) — what a biometry-gated signature does and does not protect against.

## 3. What this is not

- **Not a new authorization mechanism.** The §7 protocol shape — challenge fields, drift re-derivation, nonce store, error catalog, exit codes, and the privileged **op set** — is unchanged. SP-3 changes *which key algorithms the existing flow verifies*, *where the approver key lives*, and (the one deliberate permission-model change, §7.1) *which signers may be granted the two os-presence quarantine ops*. Nothing about what operations exist or how drift is detected changes.
- **Not a change to the audit stream.** `refs/audit/runs`, the WORM anchor, and every `SignedEnvelope` stay Ed25519-only under the attestation identity. `atlas-jcs-v1` byte-identity is untouched (the registry file is Zod-parsed JSON, never canonically signed; the response `signature` is a string value like any other).
- **Not Touch-ID-gating the daemons, the CLI, or reads.** Only authorization *signing* gains biometry. Shared commands, the brokers, and `watch` (SP-1) are untouched.
- **Not retiring Ed25519.** File-key approvers, the test signer, and Linux hosts keep working. `p256` is additive.
- **Not the Console.** SP-2 owns the GUI, its install, and how it shells out to `atlas-signer`. SP-3 defines the signer contract SP-2 consumes and refuses SP-2 any other signing path.
- **Not multi-user or remote approval.** One operator, one Mac, per-device keys. No approval quorums, no remote signers, no key sync — a second Mac enrolls its own key.
- **Not a `commands.json` change.** `--export-challenge`/`--authorization` already exist on every privileged command; `atlas-signer` lives outside the `brain` binary; enrollment is provisioning-owned. The **command** registry, command fixtures, and command schemas (`commands.json`, `cli-surface.fixture.txt`, `<cmd>.schema.json`) are untouched in SP-3 (one candidate follow-up row is R3, §12). The *signer* registry model, the authorization/contract Zod schemas, and the test-signer fixture **do** change — §§5.1–5.2; the sentence above scopes only the CLI-contract artifacts.

## 4. Scope declaration (binding for review)

Single operator, single Mac (Apple silicon, Touch ID present), personal playground install; Linux/CI hosts never see a Secure Enclave and use the software fixture path. The adversary considered is a **compromised unprivileged agent** (the existing broker threat model) plus, honestly assessed but not fully defeated, a **compromised operator GUI session** (§9). Findings that assume fleet key management, HSM ceremony, multi-approver workflows, or remote attestation are out of scope by this declaration.

## 5. Envelope & verifier changes (the TypeScript side)

### 5.1 Contracts (`@atlas/contracts`)

- `SignerRegistryEntrySchema` (`audit.ts`) gains `alg: z.enum(["ed25519", "p256"]).optional()` — **absent means `ed25519`**, so every existing `signers.json` and every key-file-derived entry parses identically — and `presence: z.boolean().optional()` (absent means `false`; the §7.1 enrollment-time custody claim). `publicKey` widens from `Ed25519PubKey` to accept the P-256 forms: SPKI PEM (`-----BEGIN PUBLIC KEY-----`) or `p256:<base64url(DER SPKI)>`. The existing `Ed25519PubKey` prefix-leniency precedent carries over: shapes are validated at parse/use time in the broker, not by regex in the schema.
- `AuthorizationResponseSchema` (`authorization.ts`): `signature` widens from `Ed25519Sig` to a prefix-discriminated union `AuthzSignature = Ed25519Sig | P256Sig`, where `P256Sig` checks the `p256:` prefix (same leniency posture).
- No other contract type changes. `IntendedEffect`, the challenge schema, and `signingPayload` construction are byte-identical to today.

### 5.2 Broker (`@atlas/broker`)

- `crypto.ts` gains the P-256 twins: `parseP256PublicKeyFlexible` (PEM or `p256:` DER SPKI → `KeyObject`, rejecting non-EC/non-P-256 keys), and `verifyP256Bytes(data, signature, publicKey)` = prefix check → base64url decode → **length bound ≤ 72 bytes** → `crypto.verify("sha256", data, publicKey, sig)` with default DER `dsaEncoding`, every failure returning `false` (fail-closed, mirroring `verifyBytes`).
- `keys.ts` `loadSignerRegistry`: entries from `signers.json` parse their `publicKey` per their `alg`. Key-file *derivation* is unchanged and stays Ed25519-only — SE keys have no broker-readable private-key file, so a `p256` signer can **only** enter the registry through explicit `signers.json` enrollment (§7). This is deliberate: hardware signers are enrolled, never inferred.
- `authorize.ts`: the verification order is unchanged (schema → canonicalization → payload recompute → nonce validate → signer registry → D20 gate → **signature** → state drift → nonce consume last). The signature step dispatches on the resolved entry's `alg`; a signature whose prefix does not match the enrolled `alg` is `authz.signature_invalid`. **No new error codes** — `signer_unknown`/`signer_revoked`/`signer_not_permitted`/`signature_invalid`/`payload_mismatch` already partition every new failure.
- **D20 widens to a set:** both `authorize.ts`'s `TEST_SIGNER_IDS` and `tools/test-signer.ts` derive their fixture ids **and fixed fixture public keys** from a **single shared descriptor** (one exported constant, not two hand-kept lists) so a fixture can never be renamed in one place and slip the production reject in the other; the resolved set — `atlas-test-approver`, `atlas-test-approver-p256` — is hard-rejected unless `ATLAS_TEST_MODE=1`, with the same precise `d20` refusal detail. `tools/test-signer.ts` gains `--alg p256` — it signs `signingPayload` (DER output) with the descriptor's committed P-256 fixture **private** key, so the in-process `BrokerService` verifies against the matching **public** key the same descriptor carries; CI exercises the entire p256 verify path with zero Apple dependency and no key exchange. The launcher still never sets `ATLAS_TEST_MODE` (provisioning invariant, unchanged).

### 5.3 Contract-doc amendment (same change, docs-with-code rule)

**SSOT after merge:** `security-broker-contract.md` is the single **operational** contract the code is lint-gated against; **this spec is the design record** (paired with immutable ADR-0002) that carries the rationale and the one-time migration plan. Where the two overlap (the `alg`/`presence`/`p256` rules, the signature-string set, the os-presence row semantics), the amended contract wins post-merge and this document is not re-edited to track it — exactly the design-spec-vs-contract split the repo's doc conventions define. The amendments below are what makes the contract carry those rules so nothing normative lives only here.

The SP-3 implementation PR amends `security-broker-contract.md` so no Ed25519-pinned prose survives where the two-member set now applies: §7.2's response example set gains a `p256:` variant; §7.3's `authz.signature_invalid` meaning is reworded alg-neutrally ("signature verification failed" — today it says "Ed25519 verification failed"); §7.4's intro drops "the non-interactive Ed25519 challenge/response" for "the non-interactive signed challenge/response", and its two os-presence rows are amended per §7.1; §8.1 names the two-member signature-string set for *authorization responses* (audit/anchor envelopes stay Ed25519-only, stated in place); §9.2 documents `alg` **and** `presence` alongside the widened `publicKey`, plus the enrollment-only rule for `p256` — and reconciles its example `permittedOps` (8 ops today) with the code's `SIGNATURE_AUTHORIZABLE_OPS` (9 ops, including `git refresh` — a pre-existing example↔code mismatch this amendment closes); §10 gains the biometry-invalidation note. In the §7.5 `authzContract` block the **ops, drift codes, and exit codes are untouched** (so `checkAuthzContractCompleteness`'s bijection and catalog checks are unaffected — it requires only non-empty `verificationSteps`), while the four `verificationSteps` strings reading "Ed25519 signature verifies over the recomputed signingPayload" are reworded alg-neutrally ("the enrolled signer's algorithm verifies over the recomputed signingPayload"). `contracts.authorization.test` gains the new example fixtures.

## 6. `atlas-signer` — the signing tool (the Swift side)

**One signing path.** `atlas-signer` is the only component that ever touches the SE key. The Console (SP-2) shells out to it; the terminal flow uses it directly; neither the Console nor `brain` ever links the key material. Recommendation R1 (§12) records why standalone-CLI beats app-embedded.

- **Placement & build:** `console/signer/` (Swift Package, macOS-only, executable target `atlas-signer`). Builds with `swift build` under Command Line Tools — no Xcode, no `.app` bundle, no entitlements, no provisioning profile; the linker's automatic ad-hoc signature suffices because the key custody avoids the data-protection keychain entirely (below). `console/` stays outside the pnpm workspace globs; CI does not build it (§11).
- **Key custody:** CryptoKit `SecureEnclave.P256.Signing.PrivateKey`, persisted as its `dataRepresentation` **blob file** at `~/Library/Application Support/atlas-signer/approver.key` (`0600`, dir `0700`) beside a small `config.json` (`signerId`, `accessPolicy`, `createdAt`, `publicKeyPem`). The blob is an SE-wrapped ciphertext usable only by this Mac's enclave — file perms are defense-in-depth, not the boundary. It lives in the **operator's** home: the agent UID cannot read it, satisfying "separately held key the agent process cannot read" structurally (and even reading it yields nothing without the enclave + biometry).
- **Access control:** `SecAccessControlCreateWithFlags(…, .privateKeyUsage | .biometryCurrentSet)` — biometry-only, invalidated on fingerprint re-enrollment (§7.3). This is the **only** V1 access policy; the scope declaration (§4) is a single Mac with Touch ID present, so V1 does not ship the `.userPresence` (biometry *or* account password / Apple Watch) fallback — it is deferred to a real Touch-ID-unavailable use case (§11). The policy is recorded in `config.json` (fixed at `biometryCurrentSet` in V1) and echoed by `pubkey`.
- **Subcommands:**
  - `keygen [--signer-id <id>]` — creates the SE key under `.biometryCurrentSet` (Touch ID prompt to prove the gate works), writes blob + config, prints the SPKI PEM and the chosen `signerId` (an explicit `--signer-id` must carry the `-vN` suffix — rejected otherwise — so `--force` rotation can always derive `-v(N+1)`). Default id: `approver-se-<hostname>-v1` on first use; **with `--force` (rotate/re-key), the default is derived from the previous `config.json`'s id with the `-vN` suffix bumped to `-v(N+1)`** — the operator never computes N by hand, and the printed enrollment line (§7) carries the new id plus the `--revoke` reminder for the old one. Refuses to overwrite an existing blob without `--force`.
  - `pubkey [--out <path>]` — writes the SPKI PEM **alone** to stdout (or `--out <path>`) and the `signerId` to stderr, so a redirect yields a valid PEM-only file for `enroll-signer.sh --pubkey` (the signer id travels on the separate channel).
  - `sign [--out <path>]` — reads one `AuthorizationChallenge` JSON on stdin; validates shape; **displays the approval summary** (below); pre-arms one `LAContext` via `evaluatePolicy` with the summary as `localizedReason`; on success constructs the key from `dataRepresentation` + that context, signs the UTF-8 bytes of `signingPayload` (`signature(for:)`, DER), **invalidates that `LAContext` immediately after the single signature** (defence-in-depth for the honest process — it cannot bind other same-UID code, §9), and emits `{schemaVersion: 1, challenge, signature: "p256:…", signerId}` as the `AuthorizationResponse`. **Channel contract (SP-2 parses this):** the approval summary and every diagnostic go to **stderr**. **Response destination — exactly one of two, never both:** with **no `--out`**, the response JSON is the *only* thing on **stdout** (success) and stdout is **empty** on failure; with **`--out <path>`**, the response JSON is written *only* to that file (`0600`, refusing an existing path unless the caller passes `--force`, failing nonzero if unwritable) and **stdout stays empty on both success and failure**. Either way the documented `> authorization.json` redirect (or `--out authorization.json`) can never capture display text. **Exit codes** (the Console branches on these, no stderr string-matching): `0` signed · `2` malformed/invalid challenge · `3` expired `expiresAt` (checked before prompting — never ask for a touch that cannot succeed) · `4` user cancelled or biometry failed · `5` key invalidated by biometry re-enrollment (stderr carries the §7.3 re-enroll runbook pointer) · `1` internal fault. (These are `atlas-signer`'s own codes — a separate binary, deliberately not the `brain` EXIT map.)
- **The approval display** — the informed-approval half the design SSOT demands (presence alone "never [counts] as approval of an unseen effect"): `sign` renders, from the challenge itself, the `op`, `runId`/`targetCommit` when present, `canonicalBaseCommit`, every `intendedEffect` field, `expiresAt`, and the SHA-256 of `signingPayload` — then puts `op` + the digest's first 8 hex into `localizedReason`, so the **system-owned** authentication sheet itself names what is being approved. **Rendering is control-character-safe:** every challenge-derived string is length-bounded and emitted in a quoted form with ANSI/C0 escapes, carriage returns, and newlines rendered visibly (never interpreted), so an attacker-influenced field value (a path, a label) cannot erase or rewrite the surrounding summary. The signer independently **re-derives** the expected `signingPayload` from the parsed challenge fields it just displayed (the same §8.2 canonicalization) and **refuses (exit 2) on any mismatch** before prompting — so the bytes it signs are provably the bytes it showed. This closes the display↔signature split a compromised GUI could otherwise exploit (send benign display fields carrying the original malicious payload, then restore the original challenge before submit): the signer never signs a payload that disagrees with its own render, and the broker's recompute-and-compare against the submitted challenge remains the second, independent backstop. (This is a deliberate departure from test-signer verbatim-signing, §8.2, which has no human to mislead.)
- **No network, no sockets, no brain invocation.** stdin → display → SE → stdout. Everything else (minting the challenge, submitting the authorization) stays in `brain`.

The terminal flow, end to end (identical shape to today's test-mode flow):

```bash
brain purge --note n_9f2c… --apply --export-challenge > challenge.json
atlas-signer sign < challenge.json > authorization.json     # displays effect, Touch ID
brain purge --note n_9f2c… --apply --authorization authorization.json
```

## 7. Enrollment, revocation, re-enrollment (provisioning)

Enrollment installs a public key into broker-owned custody, so it is **provisioning-owned and `sudo`-gated** — consistent with "the one human-led, sudo-requiring step" charter. New script `provisioning/enroll-signer.sh` (idempotent, `ATLAS_DRY_RUN`-aware, same `lib.sh` discipline):

- `enroll-signer.sh --pubkey <pem-file> --signer-id <id> --alg <ed25519|p256> [--presence]` — validates the PEM parses as the named algorithm's SPKI (both algorithms enrollable — see the break-glass note below), then **merges** an entry into `<keysDir>/signers.json` (creating it as the full derived registry + the new entry if absent — an explicit `signers.json` *replaces* derivation, so the script materializes the derived entries first rather than silently dropping the attestation/test signers). **Identity rules:** keys are compared by **DER SPKI fingerprint** (so the PEM and `p256:<base64url(DER)>` encodings of one key compare equal), never by string. Fingerprints are **globally unique across active entries**: a key already enrolled under a *different* active `signerId` is **refused** (no aliasing — otherwise a response could be relabelled with a second id to inherit that entry's `presence`/`permittedOps`/`status`, since the signature binds the payload, not the id). An existing `signerId` with a *different* fingerprint is likewise **refused** with the §7.2 rotation hint (enroll `-v(N+1)`, revoke the old id — never a silent key swap under a live id); re-running with the identical fingerprint, `alg`, `presence`, and `permittedOps` is a no-op (idempotent). A same-id, same-key re-run whose `--presence` (hence `permittedOps`) or `status` **differs** is refused with the same rotation hint — never a silent authorization-rights change under a live id. **`--presence` requires `--alg p256`** — the script **rejects `--presence` on `--alg ed25519`** (a file key proves custody, not OS-mediated presence, §7.1), closing the one-command misconfiguration that would otherwise grant an ambient file key the two quarantine ops. `permittedOps` = `SIGNATURE_AUTHORIZABLE_OPS`, plus the two quarantine ops **iff `--presence`** (§7.1). `presence` is recorded exactly as the operator passed it — an explicit enrollment-time claim, never inferred from `--alg`. Owner `atlas-broker`, mode `0600`, and a new `keys.acl.json` row so `provisioning-acl.test.ts` pins it.
- `enroll-signer.sh --revoke --signer-id <id>` — flips `status: "revoked"` + `revokedAt` (never deletes; §10 of the contract — historical verification keeps the key). Revoking an already-revoked id is a **no-op that preserves the original `revokedAt`**; an unknown `signerId` fails nonzero and mutates nothing.
- **Break-glass Ed25519 stays enrollable, not assumed.** Once an explicit `signers.json` exists, key-file derivation is bypassed permanently — so filling `approval-verify.pub` *after* SE enrollment does nothing by itself; the file-key approver must be enrolled through the same script (`--alg ed25519`, no `--presence`). R2's `doctor` check flags the trap: a non-empty `approval-verify.pub` with no matching active entry in an explicit `signers.json`.
- **The broker loads the registry at startup** — the script's last step **restarts the broker daemon** — failing loudly if it cannot, since the merged entry stays inert until reload — and prints the resulting registry summary. Enrollment is not live-reloaded; that is acceptable for an event that happens per-device-per-key-generation (R4, §12).

**Fresh-Mac story (one command per side of the trust boundary, by design):** `console/signer/install.sh` = build (`swift build -c release`) + link into `~/bin` + `keygen` + `pubkey --out approver.pem` + print the exact `sudo provisioning/enroll-signer.sh --pubkey approver.pem --signer-id <id> --alg p256 --presence` line to run. Two commands total for the operator — the second one *must* stay a separate, sudo-gated, human-typed step because it crosses from operator-space into broker custody; folding it into an unprivileged installer would let anything running as the operator enroll a signer silently. (The Console app's own install is SP-2; it requires `atlas-signer` to exist and enrolled, nothing more.)

### 7.1 The quarantine ops (`os-presence`) — resolved, not fudged

The contract authorizes `quarantine inspect`/`quarantine resolve` by **`os-presence`** — "an OS-mediated presence assertion bound to the exact broker challenge" — and `keys.ts` deliberately excludes them from every *file-key* signer's `permittedOps`, because a file signature proves key custody, not presence. Consequence today: the os-presence mechanism has **no production implementation** — the shipped `quarantine resolve` path submits a signed authorization that no enrollable signer is permitted to produce.

An SE signature released only after an OS-mediated per-use ceremony **is** a presence assertion bound to the challenge: LocalAuthentication + the enclave verify a live user act — a biometric under `.biometryCurrentSet` (V1's only policy) — before releasing exactly one signature over exactly this challenge's bytes. The policy qualifies because the contract's language is "OS-mediated presence assertion" and the ceremony is un-synthesizable by the agent (a deferred `.userPresence` variant, §11, would qualify on the same grounds). SP-3 therefore:

- adds `presence: true` to the registry entry schema (optional, default false); the enrollment script records it **only when the operator passes `--presence`** (§7) — it is an *enrollment-time claim about custody*, like `alg`: the broker cannot verify it remotely and does not pretend to; the operator asserting how this key is gated is the same trust act as installing `approval-verify.pub` custody today;
- permits the two quarantine ops in a signer's `permittedOps` **only when** `presence: true` — plain file keys stay excluded, for the reason `keys.ts` already gives, with that comment **reworded** in the implementation to say "presence-gated signers excepted" rather than preserved verbatim (verbatim, it would falsely deny what this spec adds);
- amends the contract's two §7.4 os-presence rows: the challenge-bound, presence-gated signature is the implementing assertion, and the rows' codes gain the signature-path set (`signer_*`, `signature_invalid`, `payload_mismatch`, `nonce_*`). **The refusal for a non-presence signer attempting these ops is `authz.signer_not_permitted`** — the registry step in the existing fixed verify order, exactly like any other unpermitted op. `authz.presence_unverified` stays in the catalog (codes are stable, never removed) but is documented as **reserved and unemitted**: nothing in the codebase emits it today (it exists only in the catalog and `errors.ts`), and this spec deliberately does not add an emitter — a hypothetical future *interactive* presence path would claim it.

This closes the "quarantine resolve is authorized by a mechanism nothing implements" gap with the same key and flow as everything else — one signer, one tool, nine + two ops — and with zero changes to the Authorizer's verify order.

### 7.2 Signer-id & rotation conventions

`approver-se-<hostname>-vN`, monotonic `N` per device — `keygen` **rejects an initial `--signer-id` that lacks the `-vN` suffix**, so automatic `--force` rotation (§6) always has a version to bump. Rotation = enroll `-v(N+1)`, revoke `-vN` (contract §10 verbatim — overlap window allowed for in-flight challenges). A second Mac is simply another hostname enrolling `-v1`. The Ed25519 `approval-verify` file-key slot remains supported as a break-glass approver (its custody tradeoffs are §9's first row); an operator who wants single-custody removes/never-fills the placeholder and runs SE-only.

### 7.3 Biometry re-enrollment (the invalidation story)

Apple invalidates `.biometryCurrentSet`-protected material when fingerprints are added/removed. The failure surfaces as `atlas-signer sign` exit 5; its stderr states the cause and the recovery: `atlas-signer keygen --force` (new key; the `-v(N+1)` id is derived automatically from `config.json`, §6) → two `sudo enroll-signer.sh` invocations (enroll the new id, then `--revoke` the old). Nothing else in the system is touched — the vault, ledger, and audit chain are all indifferent to which enrolled signer approves (in-flight challenges are voided by the enroll-time broker restart, §8 — re-export any that were pending). This is treated as a *feature* (a changed biometric set is exactly when silent key continuity would be wrong), and the runbook lands in `docs/install.md` with the SP-3 implementation. (A deferred `.userPresence` key, §11, would survive re-enrollment — one reason it is worth revisiting if the invalidation cost ever bites.)

## 8. Nonce TTL & flow fit

The nonce store already supports per-op TTLs (`issue(op, ttlSeconds)`, default 300 s) and stays in-memory (broker restart voids challenges — unchanged, known). The SE flow *shrinks* wall-clock relative to the manual flow that hit the TTL on the 2026-07-17 drive: export → display → touch → submit is seconds when Console-driven, tens of seconds in a terminal. **Decision: keep 300 s; no per-op overrides introduced.** The one interaction worth pinning: `atlas-signer sign` checks `expiresAt` *before* prompting (never burn a touch on a dead challenge), and `brain`'s `--authorization` submit path already yields precise `authz.nonce_expired` (exit 6) for the operator who signed too slowly — re-export, re-sign.

## 9. Threat model — what the Touch-ID gate buys, bluntly

| Adversary / event | File-key approver (status quo option) | SE + `.biometryCurrentSet` (this spec) |
|---|---|---|
| Compromised **agent** (the design's main adversary) | Cannot read the key (other UID) — but cannot be *stopped* from using it if it ever escalates to the operator UID | Same UID barrier, **plus** even operator-UID code cannot sign without a live presence ceremony — reusable across a burst by same-UID code, not silently bypassable (see the operator-malware row) |
| Malware **as the operator**, silent bulk signing | **Defeated by nothing** — key file readable, signs forever, no human in the loop | Raised, not fully defeated: a same-UID process can read the blob and rehydrate the key, so after one accepted ceremony a caller-controlled `LAContext` may be reused across several enclave operations — the floor is one visible presence event per *burst*, not strictly per signature; the honest `atlas-signer` signs once per prompt then discards the context (§6) |
| Key **exfiltration** (backup theft, disk image, sync accident) | Real risk — the PEM is portable forever | Structurally impossible — enclave-born, non-exportable; the blob is inert off-device |
| **Stolen unlocked laptop** | Attacker signs at will | Needs the operator's live fingerprint per signing burst (`.biometryCurrentSet`, V1's only policy) |
| Compromised GUI session **spoofing the approval display** | n/a (no display step) | **Not defeated.** `atlas-signer`'s rendering and even the `localizedReason` line can lie if the binary or its inputs are attacker-controlled; the system auth sheet proves *presence*, not *understanding*. Residual: the operator approves what is displayed, and display integrity rides on the session's integrity. Mitigations, honestly bounded: the signer renders only challenge-derived fields; the broker re-derives every commitment and rejects drift (`authz.*`, §7.3–7.4) — so a lied-about display still cannot authorize a *different* effect than the challenge's; what it can do is get a *legitimate but misunderstood* challenge approved. |
| Compromised **broker** | Game over either way — it is the verifier and the ref writer. Out of scope (root-equivalent by D13). | Same |
| Biometric re-enrollment | No effect | Key invalidated → §7.3 rotation (availability cost, integrity feature) |
| `--yes`, synthesized stdin, agent-driven prompts | Never authorization (unchanged, contract §7) | Same — and the SE prompt cannot be confirmed programmatically |

Two sentences of honesty the table implies: **this raises the floor from "possession of a file" to "a live-fingerprint OS ceremony (`.biometryCurrentSet`) per privileged signing burst", and that is the entire claim.** It does not make a compromised operator session safe to approve from — it makes it *loud* (each burst of signing costs at least one visible biometric prompt the human did not initiate, on an op the broker will still drift-check); it does **not** guarantee one prompt per signature (the operator-malware row bounds the reuse).

## 10. Test-live posture & CI

- **CI (`ubuntu-latest` + `macos-15`)**: no Secure Enclave exists on either leg (GitHub's macOS runners are Virtualization.framework VMs — no SEP, no biometry). CI covers: contracts schema round-trips for `alg`/`presence`/`p256:` forms; broker verify dispatch (accept valid DER over `signingPayload`; **reject** wrong-prefix, oversized-past-72-bytes, corrupted-DER / non-DER, altered-message, and other-key signatures; **and a positive regression that a high-S `(r, n−s)` twin of a valid signature still verifies `true`** — low-S is intentionally *not* enforced on either side, §13/ADR-0002, so "reject malleated" would contradict the verifier contract; the malleability posture is "verify, never byte-compare", not "reject non-canonical"); **parser rejection** cases for `parseP256PublicKeyFlexible` — an RSA key, a P-384 (non-P-256 EC) key, malformed/truncated SPKI, and trailing-garbage base64url, in both PEM and `p256:`-DER forms, each failing fail-closed *before* signature verification, plus an `alg`↔key-shape mismatch on registry load; **D20-p256**: the same flow with `ATLAS_TEST_MODE` unset must refuse with the `d20` detail (mirroring `broker.rejects-test-signer-in-prod.test.ts`); enrollment-script ACL shape via `provisioning-acl.test.ts` (non-sudo, matrix-only), **plus a behavioral enrollment test** that runs `enroll-signer.sh` against an isolated temp `keysDir` with generated Ed25519 + P-256 fixtures and asserts the §7 contract: materializing the full derived registry when `signers.json` is absent, merge-preserving the attestation/test entries, identical-key idempotency, `signerId`↔different-key collision refusal, presence-dependent `permittedOps` (the two quarantine ops iff `--presence`), revocation metadata (`status`/`revokedAt`), `ATLAS_DRY_RUN` non-mutation, `0600`/owner mode, and that the result loads via `loadSignerRegistry`. The Swift package is **not built in CI** (macOS runner could compile it, but its only meaningful behaviors — SE keygen, biometry — cannot execute there; a compile-only job is R5, §12).
- **Live (this Mac, provisioned install)**: the SP-3 implementation PR's runbook drives the real thing once per flow — **one clean `console/signer/install.sh` run** (asserting the binary builds + links into `~/bin`, `config.json` lands `0600` with the fixed `biometryCurrentSet` policy, `pubkey --out` writes a valid PEM-only file, and the printed `sudo enroll-signer.sh …` line carries the correct signer id + `--alg p256`; the `keygen`/`pubkey`/enroll command construction is additionally unit-tested where OS interaction is abstracted), `keygen` (Touch ID fires), `enroll-signer.sh --presence` + broker restart, a real `--export-challenge` → `atlas-signer sign` → `--authorization` round-trip on a low-stakes privileged op (`source trust promote` on a fixture-grade source, then undo via `source trust revoke`), **one quarantine round-trip** (`quarantine inspect` on a seeded quarantine item — the first-ever production-signer exercise of an os-presence op), a deliberate wrong-finger cancel, an expired-challenge refusal, and a `--force` re-keygen + re-enroll to rehearse §7.3 *before* the first real biometry re-enrollment forces it. Each round-trip also asserts the §6 channel contract by observation: stdout carries *only* the `AuthorizationResponse` JSON, the approval summary and every diagnostic land on stderr, every failure path leaves stdout empty, and exit codes **2–4** each fire on their documented condition (malformed, expired, cancel/biometry-fail). **Exit 5 (biometry-invalidation) is not live-rehearsable without a real fingerprint add/remove**, so it is asserted by a golden-challenge unit test that injects an `errSecAuthFailed`/invalidated-key error from the abstracted backend, while the live runbook exercises only the `--force` rotation/re-enroll *recovery* workflow that follows an exit-5 (never the invalidation trigger itself). Verified results — including these channel and exit-code observations — are pasted into the PR (test-live rule). ## 11. Consequences

- **The empty-`approval-verify.pub` era ends.** The system's privileged surface gets its first real production signer; #60's human-gated real-copy apply becomes executable as designed (D20 test-signer stays test-only, now for two algorithms).
- **A Swift toolchain enters the repo** (`console/signer/`), outside pnpm and outside CI's build graph. The contract seam (challenge/response JSON) is covered by TS tests on both sides, but the Swift *implementation* is live-verified only — a permanent asymmetry this spec accepts explicitly (§10). SP-2 inherits the same posture, bigger.
- **`signers.json` becomes load-bearing.** Until now the registry was derived from key files; after enrollment an explicit file **replaces** derivation, so the enrollment script owns keeping the derived entries present (§7). A hand-edited `signers.json` that drops the attestation entry would brick audit verification at startup — the script's merge-not-overwrite behavior plus a `keys.acl.json` row are the guardrails; `db doctor`-level checks of registry integrity are R2.
- **The os-presence mechanism becomes implementable** (§7.1) — quarantine resolve/inspect gain their first real authorization path. The `presence: true` claim is operator-asserted custody metadata; a reviewer must accept that the broker trusts enrollment-time claims exactly as it trusts key custody today (no remote attestation in V1 — R6 records the SE-attestation upgrade path).
- **Registry entries stop being algorithm-homogeneous** — every future consumer of `signers.json` (a Console signer-list view, `doctor` checks) must read `alg` instead of assuming `ed25519:`. The optional-field default keeps old files valid forever.
- **Signature strings stop being fixed-length/deterministic** (ECDSA randomness, DER variability) — pinned as "verify, never byte-compare" in ADR-0002; no current code violates it.
- **D13/D17/D18 unchanged**; D20 *widened* (a set, same hard-reject). Provisioning gains one script + one ACL row + a broker-restart step in the enrollment runbook. The CI matrix is unchanged (no new jobs; new tests ride existing legs).
- **Foreclosed / deferred:** the `.userPresence` access policy (biometry *or* account password / Apple Watch, for Touch-ID-unavailable / clamshell setups) — cut from V1 because §4's scope is a Touch-ID-present Mac; revisit when a real Touch-ID-unavailable case appears, at which point it also earns a §4 scope line and a §9 threat-model row; keychain-persisted SE keys (would demand a paid signing identity and an app bundle — revisit only if the blob-file custody proves inadequate); ML-DSA SE signing (waits on Node verify support; the `alg` enum is where it lands); remote/multi-device approval (out of scope by charter).
- **SP-2's privileged UX is now fully constrained:** display → shell out to `atlas-signer` → submit via `brain … --authorization`. Any Console design that wants to render its own Touch ID prompt or embed the key is rejected by this spec, on §9's display-integrity grounds and the one-signing-path rule.

## 12. Open questions — each with a recommendation

| # | Question | Recommendation |
|---|---|---|
| R1 | Standalone `atlas-signer` CLI vs signing embedded in the Console app? | **Standalone CLI, Console shells out** (decided, §6 — recorded here because it was genuinely contested). Reasons: one signing code path and one TCB instead of two; the terminal flow works without the GUI installed; ad-hoc/no-entitlement custody works for a CLI, while app-embedded keychain custody pushes toward paid signing identities. **Honest scope of the isolation:** this is an *honest-code architecture convention* that shrinks and de-duplicates the signing TCB — it is **not** an enforced boundary against an operator-UID compromise. Because the Console runs as the operator, a compromised Console *can* bypass `atlas-signer` and drive the blob + CryptoKit itself (§9's operator-malware row); what it still cannot do is escape the Secure-Enclave presence ceremony — every signature it forces still costs a visible biometric event. Cost: a subprocess hop and a second Touch ID surface SP-2 must design around. |

## 13. Research appendix

Current-source research behind §5–§10 (retrieved 2026-07-18; full fact sheets in the SP-3 PR thread):

1. **Secure Enclave capability set** — SE supports only 256-bit NIST P-256 for classic asymmetric signing via the keychain/SecKey path ("The only keychain items supported by the Secure Enclave are 256-bit elliptic curve private keys", *kSecAttrTokenIDSecureEnclave*, developer.apple.com; *Protecting keys with the Secure Enclave*: "Works only with NIST P-256 elliptic curve keys"); keys are enclave-born and non-importable/non-exportable ("Not having a mechanism to transfer plain-text key data into or out of the Secure Enclave is fundamental to its security"). macOS 26 added SE ML-DSA/ML-KEM via CryptoKit (SecureEnclave.MLDSA65/87 docs; WWDC25 session 314) — noted in ADR-0002 alternatives. → §1, ADR.
2. **Access-control flags** — `.privateKeyUsage` required for a usable SE signing key; `.userPresence` = biometry **or** passcode, survives re-enrollment; `.biometryAny` biometry-only, survives; `.biometryCurrentSet` biometry-only, **"The item is invalidated if fingers are added or removed"** (all verbatim from the SecAccessControlCreateFlags pages, developer.apple.com). Touch ID hardware: built into laptops; desktops via "Magic Keyboard with Touch ID for Mac models with Apple silicon" only. On biometry-less/un-enrolled setups, biometry flags are unusable and `LAContext.canEvaluatePolicy` is the documented pre-flight; `.userPresence` falls back to password/Apple Watch (*LAPolicy.deviceOwnerAuthentication*). → §6 access-control decision, §7.3, §9.
3. **Entitlement two-path split (the custody-deciding fact)** — keychain-persisted SE keys require the data-protection keychain, whose access groups are restricted entitlements needing a provisioning profile and app-like bundle (TN3137 *On Mac keychains*; Apple DTS "Quinn" threads 125510/720825: `errSecMissingEntitlement` for bare CLIs); CryptoKit SE keys are ephemeral objects whose `dataRepresentation` blob "can only be unwrapped by your SE" and may be stored anywhere (Quinn, thread 708749; *Storing CryptoKit Keys in the Keychain*: the SE "exports an encrypted block that only the same Secure Enclave can later use"). Shipping proof of the entitlement-free path: **age-plugin-se** (github.com/remko/age-plugin-se) — a plain SwiftPM CLI storing SE key blobs in files, installed via `brew`/`make`, no profile, no Developer ID. Secretive's FAQ confirms the keychain-path consequence (custom builds can't read the prebuilt app's keys) and the P-256-only limit. → §6 custody, R1, ADR alternatives.
4. **Signature-encoding interop (verified live on this Mac)** — Apple `SecKey.h`: `.ecdsaSignatureMessageX962SHA256` output "is in DER x9.62 encoding" with SHA-256 applied to the message internally; CryptoKit `ECDSASignature` exposes both `rawRepresentation` (64-byte r‖s) and `derRepresentation`; Node ≥ 24 `crypto.verify` `dsaEncoding` defaults to `'der'` ("DER-encoded ASN.1 signature structure", nodejs.org crypto docs); empirical run: SecKey and CryptoKit DER signatures verified `true` under default-option Node `crypto.verify("sha256", …)` with SPKI-PEM keys; DER length observed 70–72 bytes across runs; a malleated `(r, n−s)` still verifies (no low-S enforcement anywhere in the pair) — hence ADR-0002's verify-never-byte-compare consequence; encoding mismatch fails *soft* (`false`, not a throw) — hence pinning DER↔DER explicitly. SPKI PEM chosen as the interchange (one-liner both sides; Node 24.15+ can also import raw x963 via `raw-public`, not needed). → §5.2, ADR.
5. **LAContext flow** — keychain/SE operations drive LocalAuthentication automatically; `localizedReason` "appears in the dialog title" on macOS (`kSecUseOperationPrompt` deprecated); a pre-authenticated context passed via `init(dataRepresentation:authenticationContext:)` (or `kSecUseAuthenticationContext`) lets the subsequent signing call proceed without a second prompt — the pattern Secretive ships for its "persist authorization" feature. → §6 `sign` sequencing.
6. **CI reality** — GitHub-hosted macOS runners (`macos-15`, `macos-26`, arm64 M-series) are Anka VMs built on Apple's Virtualization.framework (actions/runner-images Packer templates; GitHub larger-runners doc: "Nested-virtualization is not supported due to the limitation of Apple's Virtualization Framework"); Virtualization.framework guests have no Secure Enclave (Apple's *Using iCloud with macOS VMs* derives VM identity "from security information in the **host's** Secure Enclave"; corroborating community analyses: khronokernel on VM attestation failure, Eclectic Light on VM FileVault ≈ "Intel Macs without T2"). No biometry in guests; `.biometryCurrentSet` items cannot be created/used without enrolled biometry. Production pattern for testability: protocol-abstracted software `P256.Signing.PrivateKey` substitution (Apple's CryptoKit keychain sample; age-plugin-se's `DummyCrypto`; facebook-ios-sdk's DPoP manager — which also documents `SecureEnclave.isAvailable` returning a false positive on Apple-silicon *simulators*, hence "probe by creating a key" if the Swift side ever needs a runtime gate). Runner-label horizon: `macos-15` + `macos-26` current; `macos-14` dies 2026-11-02; Intel macOS ends Fall 2027. → §10.
7. **Fresh-Mac install posture (shared with SP-2, cited here for the signer's slice)** — Gatekeeper on Sequoia/Tahoe removed the right-click-Open bypass; unsigned quarantined apps get no "Open Anyway" at all, ad-hoc quarantined binaries surface as "damaged" (Apple Support mh40616; MacRumors/mjtsai coverage; swissmacuser.ch Tahoe reports) — **but locally built binaries carry no quarantine xattr and launch friction-free** (quarantine is applied by downloaders, not compilers; Eclectic Light *Living with(out) notarization*, which recommends exactly this for personal software). Homebrew 5 deprecated `--no-quarantine` and will disable Gatekeeper-failing casks in the main tap from 2026-09-01 (brew.sh 5.0.0 notes) — personal taps/formulae building from source are unaffected. `swift build` works under Command Line Tools alone; `xcodebuild` requires full Xcode; SwiftPM emits no `.app` bundle (fine — the signer is a bare executable; the Console's bundle assembly is SP-2's problem). → §6 build/install, §7 fresh-Mac story.
