# Atlas SP-3 `atlas-signer` — implementation plan

> **Status:** proposed · **Epic:** [#290](https://github.com/21StarkCom/Atlas/issues/290) · **Closes:** #286, #272
> **Spec (blueprint, do not re-open):** [`docs/specs/2026-07-18-console-se-authorization-spec.md`](../specs/2026-07-18-console-se-authorization-spec.md) · **ADR:** [`docs/adr/0002-p256-secure-enclave-authorization-signer.md`](../adr/0002-p256-secure-enclave-authorization-signer.md)
> **Operational SSOT the amendment lands in:** [`docs/specs/security-broker-contract.md`](../specs/security-broker-contract.md) §7–§10.

This is an execution plan, not a spec. The spec §5–§10 is near-implementation-grade; this decomposes it into **six PR-gated, contracts-gate-first phases** matching the prior Atlas/console phase shape. Every phase lands on a branch off `feat/atlas-signer`, opens a PR, keeps `pnpm -r build && pnpm -r test && node tools/gen-cli-contract.ts --check` green (Swift phases add `cd console/signer && swift build && swift test`), posts every review finding on the PR, and updates docs-with-code.

## Global invariants (hold for every phase)

- **Additive, never subtractive.** `alg` absent ⇒ `ed25519`; every existing `signers.json` entry, key-file-derived entry, and Ed25519 flow parses and verifies byte-identically. Ed25519 is **not** retired.
- **No new error codes.** A prefix/alg mismatch, malformed body, or failed ECDSA verify is `authz.signature_invalid`. The broker verify **order is unchanged** — only the signature step dispatches on the resolved `alg`.
- **D20 stays intact and widens to a set.** No fixture signer of any algorithm authorizes outside `ATLAS_TEST_MODE`; the p256 fixture id joins the hard-reject set, derived from **one shared descriptor** so ids/keys can never drift between `authorize.ts` and `tools/test-signer.ts`.
- **Byte-identity is the whole game.** The signer signs the exact UTF-8 bytes of `signingPayload`; the broker recomputes via `buildSigningPayload` and compares. DER↔DER, SPKI-PEM keys, zero transformation at the seam.
- **No SE in CI.** SE + Touch ID is live-verified on the operator's Mac only; CI exercises the whole p256 verify path via software P-256 fixtures. `console/signer/` stays outside the pnpm workspace globs.
- **Verify-never-byte-compare.** ECDSA signatures are randomized + DER-variable; nothing byte-compares or dedupes an authorization signature. A high-S `(r, n−s)` twin of a valid signature must still verify `true` (no low-S enforcement on either side).

## Phase 1 (P1) — contracts: `alg`/`presence` + the signature union

**Package:** `@atlas/contracts`. Contracts gate — lands the types the broker and signer both bind to.

- `primitives.ts`: add `P256Sig` (`^p256:` lenient, mirroring `Ed25519Sig`) and `P256PubKey` (`^p256:`); add an `AuthzSignature = z.union([Ed25519Sig, P256Sig])` (prefix-discriminated; both are `^…:` regexes so a plain union is fine — order does not matter, shapes validated at use time in the broker). Add a `PublicKeyString` schema accepting the Ed25519 native form, the `p256:` native form, **or** an SPKI PEM (`-----BEGIN PUBLIC KEY-----`); keep the lenient posture (no DER parse in the schema).
- `audit.ts` `SignerRegistryEntrySchema`: add `alg: z.enum(["ed25519","p256"]).optional()` (absent ⇒ ed25519) and `presence: z.boolean().optional()` (absent ⇒ false); widen `publicKey` from `Ed25519PubKey` to `PublicKeyString`.
- `authorization.ts` `AuthorizationResponseSchema`: widen `signature` from `Ed25519Sig` to `AuthzSignature`.
- **No other contract type changes.** `IntendedEffect`, the challenge schema, and `signingPayload` construction are byte-identical.

**Tests (P1):** schema round-trips for an `alg:"p256"`+`presence:true` entry, an SPKI-PEM `publicKey`, a `p256:` native `publicKey`, and a `p256:` `AuthorizationResponse.signature`; a negative for `alg:"p384"`; an absent-`alg` entry still parses and infers ed25519 semantics downstream (asserted in P2). `contracts.operations`/seam determinism unaffected (no canonical-shape change).

**Done when:** `pnpm --filter @atlas/contracts test` green; the widened schemas exported from the barrel.

## Phase 2 (P2) — broker: p256 verify + alg-dispatch + D20 set + `test-signer --alg p256`

**Package:** `@atlas/broker` (+ `tools/test-signer.ts`). The security core.

- `crypto.ts`: add `parseP256PublicKeyFlexible(s)` — accepts SPKI PEM or `p256:<base64url(DER SPKI)>` → `KeyObject`, **rejecting non-EC and non-`prime256v1` keys** (check `asymmetricKeyType === "ec"` and `asymmetricKeyDetails.namedCurve === "prime256v1"`); and `verifyP256Bytes(data, signature, publicKey)` = `p256:` prefix check → base64url decode → **length bound ≤ 72 bytes** → `crypto.verify("sha256", data, publicKey, sig)` (default DER `dsaEncoding`), **every failure returns `false`** (fail-closed, mirroring `verifyBytes`). Add `serializeP256PublicKey` (→ `p256:<base64url(DER SPKI)>`) + a `generateP256` fixture-keypair helper for tests/test-signer.
- `authorize.ts`: replace the single `TEST_SIGNER_ID` with a **shared fixture descriptor** exporting `{ ed25519: {id, ...}, p256: {id, publicKey} }` and a derived `TEST_SIGNER_IDS` set (`atlas-test-approver`, `atlas-test-approver-p256`). The D20 gate rejects membership in that set outside test mode, same `d20` detail. The signature step resolves the entry's `alg` (absent ⇒ ed25519) and dispatches: ed25519 → `verifyBytes`, p256 → `verifyP256Bytes`. **A signature whose prefix ≠ the enrolled `alg` is `authz.signature_invalid`** (checked in the dispatch, not the schema). Verify order otherwise unchanged. Reword the `signature_invalid` detail alg-neutrally.
- `keys.ts`: `loadSignerRegistry` parses each `signers.json` entry's `publicKey` per its `alg` at load (via the new p256 parser for p256 entries, existing flexible parser for ed25519). Key-file **derivation** stays Ed25519-only (SE keys have no broker-readable private key → a p256 signer can only enter via explicit `signers.json`). The `permittedOps` for a presence-gated signer includes the two quarantine ops; reword the `keys.ts` os-presence exclusion comment to "presence-gated signers excepted". **The p256 fixture entry is registered UNCONDITIONALLY** — from the shared descriptor's fixed public key, exactly as `deriveSignerRegistryFromKeyFiles` registers the ed25519 fixture regardless of mode (`keys.ts:143-152`) — so `authorize.ts`'s widened D20 set produces the precise `d20` refusal, **never `authz.signer_unknown`**, when `ATLAS_TEST_MODE` is unset. (Plan-review finding: don't gate fixture *registration* on test mode; only the *authorization* is D20-gated.)
- `authorize.ts` signer-registry construction: parse a p256 entry's key via `parseP256PublicKeyFlexible`, ed25519 via `parsePublicKey`/flexible — so `Authorizer` holds the right `KeyObject` per `alg`.
- **§7.1 presence gate:** permit `quarantine inspect`/`quarantine resolve` in a signer's `permittedOps` **only when `presence: true`**; a non-presence signer attempting them is refused at the existing registry step as `authz.signer_not_permitted` (no new code, no verify-order change).
- `tools/test-signer.ts`: add `--alg p256` — signs `signingPayload` (DER output) with the shared descriptor's committed P-256 fixture **private** key, emits `signature: "p256:…"`, `signerId: atlas-test-approver-p256`. Ed25519 path unchanged.

**Tests (P2):** accept a valid DER p256 sig over `signingPayload`; **reject** wrong-prefix, oversized-past-72-bytes, corrupted/non-DER, altered-message, other-key; a **high-S twin still verifies true**; `parseP256PublicKeyFlexible` rejects RSA, P-384, malformed/truncated SPKI, trailing-garbage base64url in both PEM and `p256:` forms, fail-closed **before** verify; an `alg`↔key-shape mismatch on registry load; `test-signer --alg p256` end-to-end authorizes against in-process `BrokerService` in `ATLAS_TEST_MODE`; the **presence gate both ways** (presence:true p256 fixture authorizes `quarantine resolve`; presence:false refused `signer_not_permitted`); **D20-p256**: same flow with `ATLAS_TEST_MODE` unset refuses with the `d20` detail (mirror `broker.rejects-test-signer-in-prod.test.ts`).

**Done when:** `pnpm --filter @atlas/broker test` + the tools contract-lint suite green; the p256 verify path fully CI-covered with zero Apple dependency.

## Phase 4 (P4) — the `atlas-signer` Swift CLI

**New tree:** `console/signer/` (SwiftPM, macOS-only, executable target `atlas-signer`; outside the pnpm workspace; CI does not build it — an optional compile job is deferred to SP-2's CI per spec R5).

- **Custody:** CryptoKit `SecureEnclave.P256.Signing.PrivateKey`, persisted as `dataRepresentation` blob at `~/Library/Application Support/atlas-signer/approver.key` (`0600`, dir `0700`) beside `config.json` (`signerId`, `accessPolicy`, `createdAt`, `publicKeyPem`). Access control `.privateKeyUsage | .biometryCurrentSet` (V1's only policy).
- **Subcommands** (spec §6): `keygen [--signer-id <id>] [--force]`, `pubkey [--out <path>]`, `sign [--out <path>]`. Signer-id convention `approver-se-<hostname>-vN`; `keygen` rejects an initial `--signer-id` without a `-vN` suffix; `--force` derives `-v(N+1)` from the prior `config.json`; refuses to overwrite a blob without `--force`.
- **`sign` sequence:** read one `AuthorizationChallenge` on stdin → validate shape → **re-derive the expected `signingPayload`** from the parsed fields (same §8.2 canonicalization) and **refuse (exit 2) on mismatch** → check `expiresAt` **before** prompting (exit 3 if expired) → render the control-safe approval summary (op, runId/targetCommit when present, canonicalBaseCommit, every intendedEffect field, expiresAt, SHA-256 of signingPayload) to **stderr** → pre-arm one `LAContext` (`evaluatePolicy`, summary as `localizedReason` with op + first-8-hex of the digest) → construct the key from `dataRepresentation` + that context → sign the UTF-8 `signingPayload` bytes (`signature(for:)`, DER) → **invalidate the `LAContext`** → emit `{schemaVersion:1, challenge, signature:"p256:…", signerId}` as the `AuthorizationResponse`.
- **Channel contract (SP-2 parses this):** summary + every diagnostic → **stderr**. Response destination is exactly one of: no `--out` ⇒ response JSON is the **only** thing on stdout (empty stdout on failure); `--out <path>` ⇒ JSON only to that file (`0600`, refuse existing without `--force`), **stdout empty on both success and failure**.
- **Exit codes (its OWN table, not the `brain` EXIT map):** `0` signed · `1` internal · `2` malformed/invalid challenge (incl. re-derive mismatch) · `3` expired · `4` cancelled/biometry-failed · `5` key invalidated by re-enrollment. Matches `ConsoleCore/ExitInterpreters.swift` `SignerExit`.
- **Rendering is control-character-safe:** every challenge-derived string length-bounded and emitted with ANSI/C0/CR/NL rendered visibly (never interpreted) — mirror the Console's `ControlSafeText` posture so an attacker-influenced field cannot rewrite the summary.
- **Abstraction seam for tests:** a protocol over the signing backend so a **software `P256.Signing.PrivateKey`** substitutes for the SE key in unit tests (no SE, no biometry) — the `age-plugin-se` `DummyCrypto` pattern.
- **`console/signer/install.sh`:** `swift build -c release` → link into `~/bin` → `keygen` (fires Touch ID) → `pubkey --out approver.pem` → print the exact `sudo provisioning/enroll-signer.sh …` line.

**Tests (P4, run locally + via the software backend):** golden-challenge tests asserting rendered fields (incl. a challenge carrying ESC/CR/NL payloads whose visible output must not alter surrounding labels), `localizedReason` content, exact stdout JSON, `--out` behavior, empty-stdout-on-failure, the re-derive-mismatch → exit 2, expired → exit 3, and the 2–5 exit-code mapping (exit 5 via an injected `errSecAuthFailed`/invalidated-key from the abstracted backend); `keygen`/`pubkey`/enroll-line command construction unit-tested where OS interaction is abstracted.
  - **Cross-implementation payload golden vectors (plan-review finding, major):** the signer's re-derive-and-refuse (§6) is an availability gate — if the Swift re-derivation diverges from the broker's `buildSigningPayload`/`effectCommitmentLines` for any op, a valid challenge is exit-2-refused on first real use. So P4 ships a **committed golden-vector fixture covering all nine `intendedEffect` kinds** (integrate, revert, erase, restore, graduate, trust, forceUnblock, quarantineInspect, quarantineResolve), each a challenge whose `signingPayload` was produced by `@atlas/broker`'s `buildSigningPayload`, and asserts the Swift re-derivation reproduces those **exact bytes** (positive per-op coverage, not just the mismatch negative). A small `@atlas/broker`-driven generator emits the fixture so drift in *either* implementation breaks the gate — this same fixture is what P6 re-anchors #272 to.

**Done when:** `cd console/signer && swift build && swift test` green on this Mac; a `console/signer/CLAUDE.md` documents the tree.

## Phase 5 (P5) — enrollment, key custody, the `doctor` check

**`provisioning/` + `apps/cli` (`doctor`).**

- `provisioning/enroll-signer.sh` (idempotent, `ATLAS_DRY_RUN`-aware, `lib.sh` discipline, `require_root`):
  - `--pubkey <pem> --signer-id <id> --alg <ed25519|p256> [--presence]` — validates the PEM parses as the named algorithm's SPKI; **merges** an entry into `<keysDir>/signers.json` (materializing the full **derived** registry first when absent, so an explicit file never drops the attestation/test entries). **Identity by DER SPKI fingerprint** (PEM and `p256:` DER of one key compare equal), globally unique across active entries: a key under a different active id ⇒ **refuse** (no aliasing); an existing id with a different fingerprint ⇒ **refuse** with the §7.2 rotation hint; identical fingerprint+alg+presence+permittedOps ⇒ no-op; a same-id/same-key run whose presence/status differs ⇒ **refuse**. **`--presence` requires `--alg p256`** (reject `--presence` on ed25519). `permittedOps` = `SIGNATURE_AUTHORIZABLE_OPS` + the two quarantine ops **iff `--presence`**. Owner `atlas-broker`, mode `0600`. Then **restart the broker** (fail loudly if it can't) + print the registry summary.
  - `--revoke --signer-id <id>` — flip `status:"revoked"` + `revokedAt` (never delete); re-revoke preserves the original `revokedAt`; unknown id fails nonzero, mutates nothing.
- `provisioning/keys.acl.json`: add a `signers.json` row (owner `atlas-broker`, `0600`, `readableBy:["atlas-broker"]`) so `provisioning-acl.test.ts` pins it.
- **R2 `doctor` registry-integrity check** (a check id, not a new command): attestation entry present; no fixture signer active outside test mode; `alg`/`presence` shapes valid; the orphaned-`approval-verify.pub` trap (non-empty file, no matching active entry in an explicit `signers.json`). **Plus a defense-in-depth invariant (plan-review nit, cheap):** flag any active entry carrying a quarantine op (`quarantine inspect`/`quarantine resolve`) in `permittedOps` without `presence: true` **and** `alg: "p256"` — closing the hand-edited / materialized-`signers.json` foot-gun the R2 check exists for, beyond trusting `enroll-signer.sh`. Positive **and** negative tests for each condition — committed to SP-3, not deferred.
- **`permittedOps` drift guard (plan-review finding):** the script writes `permittedOps = SIGNATURE_AUTHORIZABLE_OPS` (+ the two quarantine ops iff `--presence`). Since bash cannot import the 9-op TS constant, the behavioral enroll test **asserts the merged non-quarantine `permittedOps` set equals `SIGNATURE_AUTHORIZABLE_OPS` exactly** (both `--presence` and non-presence enrollments), so a bash↔TS divergence fails CI.
- `docs/install.md`: signer / enrollment / re-enrollment (exit-5 rotation runbook) section; `provisioning/CLAUDE.md` updated.

**Tests (P5):** `provisioning-acl.test.ts` pins the new row; a **behavioral enrollment test** against an isolated temp `keysDir` with generated Ed25519 + P-256 fixtures asserting the §7 contract (materialize-when-absent, merge-preserve, identical-key idempotency, **id↔different-key collision refusal, the aliasing refusal — the SAME key enrolled under a DIFFERENT active `signerId` (no relabel to inherit another entry's presence/permittedOps/status)**, the same-id/same-key presence-or-status-differs refusal, presence-dependent `permittedOps` **and its base-set equality to `SIGNATURE_AUTHORIZABLE_OPS`**, `--presence`-on-`ed25519` rejection, revocation metadata, `ATLAS_DRY_RUN` non-mutation, `0600`/owner, and that the result loads via `loadSignerRegistry`); the `doctor` positive/negative suite (incl. the quarantine-op-without-presence flag).

**Done when:** the enrollment script + ACL row + doctor check + `docs/install.md` land with green tests.

## Phase 6 (P6) — live drive + close-out (human-led)

**On the provisioned Mac (`ATLAS_PROVISIONED` layout, launchd daemons live).**

- One clean `console/signer/install.sh` run (Touch ID prompt proves the gate; asserts binary builds + links, `config.json` `0600` + `biometryCurrentSet`, `pubkey --out` writes a valid PEM, the printed enroll line carries the right id + `--alg p256`).
- `enroll-signer.sh --presence` + broker restart.
- A real `--export-challenge` → `atlas-signer sign` → `--authorization` round trip on a low-stakes op (`source trust promote` on a fixture-grade source, undo via `revoke`); **one quarantine round trip** (`quarantine inspect` on a seeded item — the first production-signer os-presence exercise); a deliberate wrong-finger cancel (exit 4); an expired-challenge refusal (exit 3); a `--force` re-keygen + re-enroll to rehearse §7.3. Each round trip asserts the §6 channel contract by observation (stdout = only the response JSON; summary + diagnostics on stderr; empty stdout on failure; exit codes 2–4 on their documented conditions).
- Drive the **SP-2 Console's full export→sign→authorize round trip** against the live broker — the thing #257 could only take up to the fail-closed refusal.
- **Spec §10 real-vault done-when (plan-review finding, major) — reconciled, not dropped.** Spec §10's final clause wants *a full privileged op on the real vault (`graduation migrate --apply` for #60's human-gated slice, or `purge`) authorized through Touch ID instead of the test signer.* SP-3 does **not** destructively `purge` real history nor run the #60 real-copy `graduation migrate --apply` — **that op is #60's human-gated slice by charter** (spec §11: "#60's human-gated real-copy apply becomes executable as designed"). So SP-3's own live drive uses **`source trust promote`/`revoke`** (reversible) + the quarantine round trip as the production-signer Touch-ID exercise, and this §10 clause is **explicitly carried by #60**: the first `graduation migrate --apply` on the real vault, authorized through the enrolled SE signer, closes it. The SP-3 retro records this reconciliation; #290's close notes the residual rides #60 (it does not silently block the SP-3 merge, since every mechanism — p256 verify, presence gate, the signer, enrollment — is proven by the reversible ops + the full CI matrix + the Console round trip).
- **Re-anchor #272:** point `SignerContractValidatorTests` at the `atlas-signer` package's committed example challenge/response fixtures (or, interim, a shared TestSupport fixture derived from the signer's golden output) so upstream contract drift breaks the gate.
- Capture `docs/retros/2026-07-20-atlas-signer-live-drive-retro.md`. Close #286 + #272 with a pointer to the merge.

**Live-drive gotchas (verify each still holds):** daemons run as launchd services (`provisioning/macos/services.sh status`); the deployment CLI identity is `atlas-agent` (D17); instance state at `~/Code/Vaults/atlas-live/.atlas`, sockets closed to the operator UID; **enrollment restarts the broker, voiding in-flight nonces** (300 s TTL) — sign + authorize promptly; export `ATLAS_EGRESS_CAPABILITY_KEY` for mint-bearing commands.

## Risks & mitigations

- **Apple→Node DER seam drift.** Mitigated by the ADR's live-verified pinning (DER default `dsaEncoding` + SPKI PEM, zero transformation) and the P2 software-fixture round trip that exercises the exact `crypto.verify` path CI-side.
- **D20 slip between two lists.** Mitigated by the single shared fixture descriptor (ids + keys derive from one constant).
- **`signers.json` becomes load-bearing** (explicit file replaces derivation). Mitigated by the enrollment script's materialize-then-merge + the ACL row + the R2 doctor check.
- **Review non-convergence (SP-2 lesson).** 2-round review cap; hand-fix wing findings (never re-dispatch); round 3 = escalate. Every reviewer proves it can run the suite before round 1.
